## 핵심 이슈 (3개)

### 1. ⚠️ `reraise` 플래그 불일치로 인한 잠재적 문제
**파일**: `pykis/account/api.py`

```python
# 주문 메서드 - reraise=True (예외 발생)
@api_method("현금 주문", reraise=True)
def order_cash(...): ...

# 조회 메서드 - reraise 미지정 (기본값 False, None 반환)
@api_method("계좌별 주문가능 수량 조회")
def get_account_order_quantity(...): ...
```

**문제점**: 
- 조회 메서드에서 오류 시 `None` 반환 → 호출자가 `None` 체크를 누락하면 `NoneType` 에러 발생
- 기존 코드는 명시적으로 `return None`을 했지만, 데코레이터 패턴에서는 암묵적

**권장사항**: 조회 메서드에 `default_return={}` 또는 적절한 기본값 명시

---

### 2. ⚠️ `exceptions.py`의 `@api_method` 데코레이터 - ExceptionHandler 미상속 시 동작
**파일**: `pykis/core/exceptions.py:280-288`

```python
else:
    # ExceptionHandler를 상속받지 않은 경우 기본 로깅
    logger = logging.getLogger(self.__class__.__name__)
    error_msg = f"[{context}] {e.__class__.__name__}: {str(e)}"
    logger.error(error_msg, exc_info=True)

    if reraise:
        raise APIException(error_msg) from e
    return default_return
```

**문제점**: 
- `ExceptionHandler`를 상속받지 않은 클래스에서 사용 시 `_exception_logger` 대신 새 로거 생성
- `BaseAPI`는 `ExceptionHandler`를 상속하므로 현재는 문제없으나, 향후 다른 클래스에서 사용 시 로깅 일관성 깨짐

**심각도**: 낮음 (현재 구조에서는 발생 안 함)
